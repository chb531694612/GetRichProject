# GetRichProject Agent 通用说明

本文件作用于整个仓库。任何 Agent 在本项目中分析、修改、测试、提交或指导部署前，都必须先阅读并遵守本文件。若子目录以后存在更具体的 `AGENTS.md`，则子目录文件只补充或收紧对应范围的规则。

## 1. 用户与沟通方式

- 默认使用简体中文与用户沟通，命令、代码、配置键保持原文。
- 把用户当作第一次接触 Python、Docker、Git 和 1Panel 的新手：说明当前目的、预期结果、失败时何时停止。
- 涉及服务器升级部署时必须“一次一步”：每次只给一个小阶段的命令，等待用户贴出完整输出并核对成功后，再给下一步。
- 不要一次性把拉取、备份、构建、迁移、启动、清理全部丢给用户执行。
- 遇到报错先依据日志和实际状态诊断，不让用户盲目重试，不宣称未验证的结果。
- 用户截图或日志中如出现密钥、授权码、Cookie、密码或私钥，应立即提醒遮蔽；不要要求用户发送敏感值。

## 2. 项目概况

- 项目：竞彩足球比分（CRS）与胜平负（HAD）串关推荐、邮件、结算及个人看板。
- 语言：Python 3.11+，生产镜像使用 Python 3.12。
- 主包：`src/score_fourfold/`。
- 测试：`tests/`，使用标准库 `unittest`。
- 数据库：SQLite，生产容器内路径 `/app/data/score_fourfold.db`。
- 运行数据：Docker 命名卷 `score-fourfold_score-data`（实际操作前仍须用 `docker inspect` 核对真实卷名）。
- 当前部署：1Panel 管理的 OpenResty 提供 HTTPS；本项目 Compose 只运行 app，不再运行 Caddy。
- app 固定容器名：`score-fourfold-app`。
- app 加入外部 Docker 网络：`1panel-network`。
- 当前 1Panel OpenResty 容器使用 `host` 网络，反向代理目标为 `http://127.0.0.1:8080`；只有将 OpenResty 加入 `1panel-network` 后才可改用 `http://score-fourfold-app:8080`。
- 站点代理片段位于 `/opt/1panel/www/sites/score-fourfold/proxy/root.conf`。AI 请求最长允许 600 秒，因此该片段必须设置 `proxy_connect_timeout 30s`、`proxy_send_timeout 660s` 和 `proxy_read_timeout 660s`，为应用处理和重定向额外保留 60 秒。
- 宿主机仅绑定 `127.0.0.1:8080`，不得在安全组或防火墙公开 8080。

## 3. 不可破坏的安全边界

- 永远不要提交 `.env`、`.env.*`、数据库、邮件预览、备份压缩包、API Key、SMTP 授权码、网页登录哈希明文来源或 SSH 私钥。
- `.env.example` 只能放安全的示例和空值，不得写入真实凭据。
- 任何数据库迁移、版本升级或高风险修改前，必须先指导用户：
  1. 停止 app 写入；
  2. 备份 `.env`；
  3. 识别真实数据卷名；
  4. 只读挂载卷并生成可检查的压缩备份；
  5. 验证备份非空且能列出内容。
- 禁止执行 `docker compose down -v`。
- 禁止在未核对目标路径或卷名时清空、覆盖、移动数据库卷。
- 删除旧容器前先用精确名称过滤并确认状态；容器、卷和目录必须分开处理。
- 保留旧代码目录和升级前数据备份至少 7 天，除非用户明确要求并再次确认删除。

## 4. 代码与数据一致性规则

- 保留公共接口兼容性。新增多计划接口时，不得无意改变旧单结果接口的返回类型。
- CRS 目标行为：按 `2串1 → 3串1 → 4串1` 尝试，最多 3 张；只有数据源明确支持相应过关方式时才生成。
- HAD 每个推荐日期最多 1 张；CRS 每个推荐日期最多 3 张。数量检查与写入必须处于原子事务中，防止并发越界。
- 概率阈值配置目前仅作历史兼容，不得重新加入单场概率或联合概率硬过滤，除非用户明确改变产品策略。
- `Recommendation.ai_summary` 必须持久化到 `plans.ai_summary`；历史计划允许为空，手动分析成功后刷新必须仍存在。
- AI API 失败不得阻断主推荐流程，但手动 AI 操作必须向用户显示明确的成功、空结果或失败提示。
- AI 提供方为阿里云百炼千问 Responses API，并强制使用联网搜索。发送给 AI 的比赛内容只能包含比赛 ID/编号、日期、开赛时间、联赛和球队；不得发送或在提示词中列出赔率、SP、概率、候选项、当前推荐或系统策略排序。
- Web 状态修改操作必须使用 POST、登录会话、CSRF、Host、Origin/Referer 和代理头校验。
- CSP 不允许全局放开脚本。内联脚本必须使用每次响应生成的 nonce；不得使用 `javascript:` URL 或未受 CSP 授权的内联事件。
- AI 手动分析必须同时生成总体摘要和每场结构化推荐；每场推荐只能引用数据库保存的真实选项，页面逐行显示，并由用户明确点击后才可替换原推荐。不得直接采用 AI 自行生成的比分代码、赔率或概率。
- 单场手动修改与采用 AI 推荐都必须使用 POST、CSRF 和来源校验，并同步重算串数、联合赔率、联合概率、理论奖金及已结算计划的实际结果；不得自动批量覆盖用户原推荐。
- 删除单场后必须同步重算：串数、联合赔率、联合概率、理论奖金、税、净奖金；若计划已结算，还必须重算实际状态、实际返还和净盈亏。
- 删除单场后清空旧 AI 摘要，因为旧摘要可能描述已删除比赛。
- 删除整张计划时同步清理其 recommendation/settlement 邮件去重记录，保证同一确定性 plan ID 以后可安全重建并重新入队。
- 当天首封购买推荐邮件不得早于北京时间 15:00；15:00 前的看板修改应刷新尚未发出的首封邮件，首封已发送后的单场删除、手动修改或采用 AI 推荐必须重新排队发送最新版推荐邮件，且仍受推荐邮件截止时间约束。
- 数据库迁移必须幂等、可从旧版本直接升级、不能丢历史计划；涉及表主键变化时必须检查真实 `PRAGMA table_info`，不能只按某个普通列是否存在判断。
- 金额统一使用 `Decimal` 和项目已有的奖金封顶、五成双舍入及税费函数，不得使用 float 或另写不一致的公式。

## 5. 修改前后的检查流程

修改前：

1. 查看 `git status --short --branch`，确认并保护用户已有的未提交改动。
2. 阅读相关实现、调用方和测试，不能只按修改报告判断“已经实现”。
3. 涉及 Web、数据库、邮件、结算时沿完整调用链检查状态联动。

修改后至少执行：

```powershell
$env:PYTHONPATH='src'
python -m compileall -q src
python -m unittest discover -s tests -t . -q
git diff --check
```

- Windows 的 LF/CRLF 提示可以说明，但不能把真正的 `git diff --check` 错误当作换行提示忽略。
- 新功能和每个已修复 bug 都必须有针对性回归测试；不得只修改旧断言来掩盖实现错误。
- 只有完整测试通过才允许提交推送；若因环境缺少 Docker，必须明确说明本地无法完成的 Docker 检查，并在服务器部署阶段补做。
- 修改 Compose 后在有 Docker 的环境运行 `docker compose config --quiet`。

## 6. Git 提交与自动推送

- 用户要求本项目修改完成后自动提交推送。完成代码或文档修改并通过相应验证后，Agent 应自动执行 Git 提交和推送，无需再让用户手工输入。
- 推送前必须检查：
  - 当前分支和远程；
  - `git diff --check`；
  - 待提交文件列表；
  - 没有 `.env`、数据库、备份或无关用户改动。
- 只 `git add` 本次任务相关文件，不得顺带提交无关改动。
- 提交消息使用清晰的英文 Conventional Commit 风格，例如 `fix: ...`、`feat: ...`、`docs: ...`。
- 默认推送当前分支到已跟踪的 GitHub 远程；当前通常为 `origin/main`。若存在阿里云效等多个远程，先核对名称，不得覆盖或误推。
- 推送成功后报告提交哈希和远程分支；推送失败时保留本地提交，说明原因和安全的恢复/重试方式。
- 不得强制推送、改写公共历史、删除远程分支或标签，除非用户明确授权并理解影响。
- 版本发布时同步检查 `pyproject.toml`、`src/score_fourfold/__init__.py`、`compose.yaml` 和客户端 User-Agent 中的版本号；普通修复提交不擅自提升版本。

## 7. 1Panel + OpenResty 升级部署流程

任何需要部署到生产服务器的变更，都按以下阶段逐步指导，并在每个阶段等待用户输出：

1. **确认现场**：项目目录、当前提交、`docker compose ps`、真实数据卷名、`1panel-network` 是否存在。
2. **备份（涉及数据或版本升级时）**：停止 app，备份 `.env` 和数据卷，验证压缩包。
3. **拉取代码**：`git pull --ff-only origin main`，再用 `git log -1 --oneline` 核对预期提交。
4. **构建**：`docker compose config --quiet` 后执行 `docker compose build app`。
5. **启动前验证**：依次运行 `check-config`、`health`；按变更需要再运行只读的 `probe-data`，启用 AI 时可运行 `probe-ai`。
6. **正式启动**：`docker compose up -d --force-recreate app`。
7. **运行验证**：检查 `docker compose ps`、最近日志、容器内 `score-fourfold health`。
8. **OpenResty 验证**：浏览器访问 HTTPS 域名，确认登录、Host/Origin 校验与反代正常。反代目标必须与 OpenResty 实际网络模式一致；当前 `host` 网络保持 `http://127.0.0.1:8080`。
9. **功能验收**：针对本次变更执行最小且非破坏性的页面验证。删除类功能不得拿真实生产计划随意测试。
10. **清理**：只有新版本稳定后才精确清理确认过的旧容器；不自动删除卷、备份或旧代码目录。

部署期间的关键规则：

- GitHub 推送不会自动更新服务器；必须明确指导服务器 `git pull`、重建和重建容器。
- 同一个镜像标签重新构建后，必须使用 `--force-recreate` 确保运行容器采用新镜像。
- OpenResty 由 1Panel 管理，不要重新添加 Caddy 服务，不要占用宿主机 80/443。
- 若出现 502，先检查 app 是否 healthy、宿主机 `127.0.0.1:8080` 是否可达，以及 OpenResty 代理目标是否与其网络模式一致，不得盲目公开 8080。
- 若长时间 AI 操作出现 504，检查站点错误日志中的 `upstream timed out`，并核对代理片段的 `proxy_read_timeout` 和 `proxy_send_timeout` 均不少于 660 秒；修改后必须先执行 `openresty -t`，成功后再平滑重载。
- 若出现操作 403，检查 `WEB_PUBLIC_ORIGIN`、Host、`X-Forwarded-Proto`、`X-Forwarded-For`、Origin/Referer 和登录 CSRF，不得通过关闭安全校验绕过。

## 8. 完成标准

只有同时满足以下条件才可以向用户报告“完成”：

- 请求的功能真实实现，不只是界面出现或报告声称已实现；
- 相关数据联动与异常路径已检查；
- 新增回归测试且完整测试通过；
- 本次相关文件已提交并成功推送；
- 如需生产部署，服务器已拉取正确提交、容器 healthy、日志无 traceback、网页和本次功能均已验证；
- 已向用户说明任何尚未在当前环境验证的项目。
